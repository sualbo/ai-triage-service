# Сценарий защиты проекта 5–7 минут

## 1. Проблема бизнеса — 1 минута

В онлайн-сервисах, интернет-магазинах и образовательных проектах обращения клиентов приходят из разных каналов: email, форма сайта, чат поддержки.

Менеджеры тратят время на первичную сортировку: понять тему обращения, определить срочность, написать первый ответ и решить, нужно ли передавать вопрос специалисту.

Из-за ручной обработки падает скорость реакции, а качество ответов становится нестабильным.

## 2. Контекст и ограничения — 1 минута

В рамках MVP реальные каналы не подключались. Канал передается как метка: `email`, `form`, `chat`.

Ограничения проекта:

- один основной endpoint `POST /triage`;
- максимум 4 категории;
- SQLite вместо сложной БД;
- без RAG и базы знаний;
- простое лимитирование;
- секреты только через env;
- при сбое LLM сервис должен безопасно передавать обращение оператору.

## 3. Решение и архитектура — 2 минуты

Я реализовал FastAPI-сервис.

Поток обработки:

```text
Запрос клиента
  -> валидация входа
  -> проверка лимита по client_id
  -> LLM-классификация
  -> guardrail-проверка JSON
  -> запись в SQLite
  -> JSON-ответ
```

Сервис возвращает:

- `category` — тип обращения;
- `draft_reply` — черновик ответа;
- `confidence` — уверенность;
- `escalate` — нужно ли передать оператору.

История сохраняется в таблицу `tickets`.

## 4. Надёжность и безопасность — 1–2 минуты

Что сделано:

1. Валидация входного JSON через Pydantic.
2. Ключ LLM хранится только в `.env`.
3. `.env` исключен из Git.
4. Добавлено лимитирование запросов на `client_id`.
5. Добавлен fallback при ошибке LLM.
6. Добавлен guardrail layer: сервис проверяет JSON от модели и не доверяет LLM напрямую.
7. Все обращения сохраняются в SQLite для аудита.
8. Основные события пишутся в лог.

Портфолио-фича — guardrail layer. Она показывает, что проект не просто вызывает модель, а учитывает реальные риски AI-сервисов: невалидный JSON, сбой API, низкую уверенность и необходимость эскалации.

## 5. Демонстрация — 1 минута

Показать:

1. Запуск сервиса.
2. POST-запрос на `/triage`.
3. JSON-ответ.
4. Запись в SQLite через `scripts/query_tickets.py`.
5. Логи в `logs/app.log`.

## 6. План внедрения за 1 день — 1 минута

План внедрения:

1. Согласовать категории обращений.
2. Подготовить OpenAI API key и env-переменные.
3. Запустить сервис локально или через Docker.
4. Проверить 20–30 реальных обращений.
5. Настроить лимиты и fallback.
6. Подключить внешний канал: форму сайта, CRM или чат.
7. Добавить просмотр истории обращений.

## 7. Что улучшить дальше

В v2 можно добавить:

- RAG / базу знаний компании;
- Redis вместо in-memory rate limiter;
- админку;
- авторизацию API;
- интеграции с Gmail/CRM/Telegram;
- мониторинг;
- аналитику качества ответов.
