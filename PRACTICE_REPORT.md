# Отчёт по практике

## 1. Кому и какую проблему решаем

Проект решает задачу онлайн-сервиса, интернет-магазина или образовательного проекта, где менеджеры получают обращения из разных каналов: email, формы сайта и чата.

Проблема: обращения приходят в разном виде, требуют первичной сортировки и занимают время менеджеров. Из-за этого замедляется реакция поддержки и сложнее контролировать качество обработки.

Решение: MVP API-сервиса, который принимает текст обращения, классифицирует его, формирует черновик ответа, определяет необходимость передачи человеку и сохраняет историю обработки.

## 2. Ограничения и допущения

Ограничения MVP:

- реализован один основной endpoint: `POST /triage`;
- реальные каналы email/form/chat не подключаются, канал передается как метка;
- используется максимум 4 категории: `billing`, `support`, `complaint`, `other`;
- база знаний / RAG не используются;
- SQLite используется как простое локальное хранилище;
- лимитирование реализовано in-memory и подходит для одного процесса;
- LLM-ключ хранится только в переменных окружения.

Допущения:

- сервис запускается локально или в Docker;
- пользователь сам предоставляет реальный `OPENAI_API_KEY`;
- для учебной демонстрации можно использовать `LLM_FAKE_MODE=true`, чтобы показать работу без внешнего API.

## 3. Архитектура

Схема блоками:

```text
POST /triage
  -> Input validation
  -> Rate limit by client_id
  -> LLM processing
  -> Guardrail JSON validation
  -> SQLite audit log
  -> JSON response
```

Fallback-сценарий:

```text
LLM error / invalid JSON / missing key
  -> category=other
  -> confidence=low
  -> escalate=true
  -> draft_reply="Ваше обращение передано оператору..."
  -> save error to SQLite
```

## 4. Контракт программного интерфейса

### Вход

```json
{
  "text": "Здравствуйте, я оплатил заказ, но доступ к курсу не открылся.",
  "channel": "email",
  "client_id": "client_001"
}
```

### Выход

```json
{
  "category": "billing",
  "draft_reply": "Здравствуйте! Спасибо за обращение. Мы проверим информацию по оплате и передадим вопрос специалисту, если потребуется уточнение.",
  "confidence": "high",
  "escalate": false
}
```

## 5. Как устроен журнал обработки

История сохраняется в SQLite в таблице `tickets`.

Поля таблицы:

| Поле | Описание |
|---|---|
| `id` | ID записи |
| `created_at` | дата и время создания |
| `client_id` | идентификатор клиента |
| `channel` | канал обращения |
| `text` | исходный текст обращения |
| `category` | категория обращения |
| `confidence` | уверенность классификации |
| `escalate` | нужно ли передать оператору |
| `draft_reply` | черновик ответа |
| `error` | текст ошибки, если был fallback |

Путь к базе по умолчанию:

```text
data/tickets.db
```

## 6. Надёжность

В проекте реализованы базовые механизмы надежности:

1. **Валидация входа**  
   FastAPI и Pydantic проверяют обязательность полей, длину текста, допустимые значения `channel` и формат `client_id`.

2. **Секреты через env**  
   `OPENAI_API_KEY` не хранится в коде и не должен попадать в репозиторий.

3. **Лимитирование**  
   На каждый `client_id` действует ограничение `REQUESTS_PER_MINUTE`.

4. **Fallback при сбое LLM**  
   Если модель недоступна, ключ не задан или ответ модели не соответствует контракту, сервис не падает, а возвращает безопасную эскалацию.

5. **Guardrail Layer**  
   Ответ LLM проверяется как строгий JSON: допустимые значения, обязательные поля, отсутствие лишних полей, 1–6 предложений в `draft_reply`.

6. **Логирование**  
   Ключевые этапы пишутся в `logs/app.log`.

## 7. Развёртывание / демонстрация

Для сдачи можно использовать путь B: публичная демонстрация без платного сервера.

План демонстрации:

1. Запустить сервис локально.
2. Отправить запрос на `POST /triage`.
3. Показать JSON-ответ.
4. Показать запись в SQLite.
5. Показать логи.
6. Показать fallback-сценарий при выключенном ключе или при `LLM_FAKE_MODE=false` без `OPENAI_API_KEY`.

Также проект подготовлен к Docker-запуску через `docker compose up --build`.
